### G05 — 認證重構：username 登入 + remote-first fallback local + 來源互斥（進行中，2026-06-09）

> 走 Secure SDLC 編排（Gate G1~G6）。計畫：`C:\Users\zzx78\.claude\plans\swift-juggling-otter.md`（已核准）。
> CoreMain 對齊：服務「組織內部 100 人協作」的登入體驗與 SSO，屬基礎建設修正，圍繞主題、不過度設計。

**問題（為什麼做）**：
1. email 同時當登入鍵與聯絡信箱 → 無法改 email；remote 帳號 email 被綁死。
2. auth_backend 單選無 fallback → 純本地帳號在 ldap/radius 模式登不進；admin-always-local hack 導致 remote 帳號升 admin 後鎖死。
3. local 與 remote 帳號可隱性重疊，auto-provision 無防呆。

**上線標準（Exit Criteria 三維度）**：
- 功能：username 登入；remote-first→fallback local；local 可改 email、remote 自動帶遠端 email；前端登入/註冊/設定改 username。
- 資安：無認證繞過；來源互斥（local/remote 同名擋下）；remote 帳號不得升 admin（防鎖死）；無硬編碼密鑰；auth_source 不可被使用者竄改。
- 品質：後端覆蓋率 ≥95%（CI gate 不退）；既有測試全綠；新分支全覆蓋。

##### 〔Sec〕資安確認結果（對照 OWASP A07 認證失效）
- ✅ **認證繞過**：login 兩段式（remote-first → fallback local），任一段失敗皆 raise 401；無預設放行路徑。
- ✅ **來源互斥**：username 全域唯一；remote 不接管 local 同名帳號（`user.auth_source==backend` 才走 remote）。register 撞任何來源同名 → 400。
- ✅ **提權**：role 變更受 require_admin 保護，無繞過。**remote 帳號可升 admin**（2026-06-09 依使用者需求開放）——因 login 不再強制 admin 走 local，remote admin 走目錄驗證不會鎖死；測試驗證升級後仍能登入。取捨：未強制保留 local admin，所有 admin 皆 remote 且目錄中斷時期間無 admin 可登入（已確認接受）。
- ✅ **密鑰**：無硬編碼；LDAP/RADIUS 密鑰走 system_settings 加密儲存（crypto.encrypt_secret）。
- ✅ **欄位防竄改**：auth_source 不在 UserUpdate schema → 使用者無法透過 PATCH /users/me 竄改來源（測試 test_user_cannot_tamper_auth_source 驗證）。
- ✅ **email 帶入**：remote 登入自動帶/更新目錄 email；RADIUS 無 email 屬預期（None）。
- **G3 過程發現並修正（真實設計缺陷）**：原 login 對「remote 成功但來源不符」直接 raise 401，導致 local 帳號在 ldap 模式下被同名 remote **鎖死**（連本地密碼都用不了）。改為「來源不符 → remote 不接管 → 落 fallback local」，互斥達成且不鎖死。見 test_source_conflict_local_user_cannot_login_via_remote。

1. AD 同步時自動帶入 AD 的 **OU 階層**（DN 路徑），展開成 org_units 樹。
2. 與**手動建立的組織樹並行**——兩套互不覆蓋。
3. 觸發：**admin 手動按鈕 + 每日自動**同步。

**設計決策（使用者確認）**：
- **樹來源 = OU 階層**：search base 下所有 `organizationalUnit`，以 DN 的父子層級組 parent_id。
- **並行隔離**：org_units 加 `source`（manual/ad）。同步**只新增/更新 source=ad**，永不動 source=manual。
- **冪等對應**：org_units 加 `external_id`（ad=OU 的 DN，manual=null），重跑同步以 external_id 找既有單位更新而非重複建。

- **讀取憑證 = 沿用現有 ldap_bind_dn/password 服務帳號**（不新增憑證、不增攻擊面）。
- **使用者歸屬**：同步只在「ad 帳號 + org_unit_id 為空或指向 ad 單位」時依 AD OU 自動帶；手動指派（指向 manual 單位）的不覆蓋。
- **觸發**：手動 `POST /org-units/sync-ad`（admin only，回傳新增/更新/停用摘要）；自動沿用 main.py lifespan 每日排程（同 auto_archive loop 模式）。

### G11 — AD 使用者同步 + 有 DN 自動歸 OU（Round 19，進行中 2026-06-12）

> 走 Secure SDLC 編排。延續 [[G10]]（OU 樹同步）與 [[G05]]（remote-first 認證）。
> CoreMain 對齊：服務「組織內部 100 人協作」，讓 AD 組織的人員與部門一鍵帶入；遵守「簡單、不過度設計」。

**需求（使用者確認）**：
1. 同步時一併撈 AD 使用者，**預建本地帳號**（帳號、姓名、Email、部門/課別、職位），**不存密碼**。
2. 使用者真正登入時走遠端 LDAP 驗證（沿用 G05 remote-first；預建帳號 = 先佔位帶資料）。
3. **判斷機制**：使用者有 DN 且其父 OU 對應到已同步單位 → 自動歸該 OU；否則只同步使用者、不指派單位。

**設計決策（使用者確認）**：

- **欄位帶入**：username=sAMAccountName、display_name、email、position=title、org_unit=DN 父 OU 對應單位。
- **帳號建立策略**：預建所有 AD 使用者（auth_source=ldap、placeholder 密碼 `__remote_auth__`），登入照走遠端驗證（與既有 auto-provision 相容——login 找到既有預建帳號即更新 email、不重複建）。
- **歸屬判斷**：user DN 的父 OU（normalize_dn）對應到已同步 ad 單位 → 設 org_unit_id；對應不到（掛在 container 或 OU 未同步）→ 不指派（org_unit_id 維持）。**不覆蓋手動指派**（org_unit_id 已指向 manual 單位者不動）。

- **User 新欄位**：`external_id`（DN，nullable，唯一性不強制——AD 重建 DN 可能變）。
